STS2 Local Model Harness Flow

A concrete map of what talks to what, what the Qwen model sees, and where legality, strategy, and logging happen in the current local setup.

Model: qwen3.5-9b LLM API: 127.0.0.1:8080/v1 Game API: localhost:15526 MCP transport: stdio

System Map

Harness is the orchestrator It fetches state, filters current-state tools, builds the model prompt, validates the chosen JSON, calls one tool, logs the result, then repeats.
MCP is the typed tool boundary It exposes schemas and tool names, then converts calls like rewards_claim(reward_index) into STS2_MCP REST actions.
llama-server only predicts It does not see the game directly and it does not call tools. It receives text messages and returns text, ideally one JSON object.

One Harness Step

Order Sender Receiver Payload Important Detail
1 Harness MCP server list_tools() once at startup Harness discovers global MCP tool names, descriptions, and JSON schemas. Profile/wiki tools and multiplayer tools are filtered out.
2 Harness MCP server get_game_state({"format":"json"}) MCP maps this to GET http://localhost:15526/api/v1/singleplayer?format=json.
3 STS2_MCP mod Harness Full state JSON Includes state_type, player data, visible cards, enemies, rewards, options, indices, and flags like can_play.
4 Harness Harness tools_for_state(state, tools) Global MCP tools become a current-state menu. Example: rewards shows rewards_claim, while card_reward shows rewards_pick_card.
5 Harness llama-server OpenAI chat completion request Messages include the system prompt, high-level strategy, current-state action menu, recent action history, and compacted current state JSON.
6 llama-server Harness Raw model text Expected shape is exactly {"action":"...","args":{...},"rationale":"..."}. The harness extracts and parses this JSON.
7 Harness Harness Validation and optional repair prompt Rejects missing args, wrong state tools, unplayable cards, bad targets, invalid indices, and illegal reward screen transitions.
8 Harness MCP server One tool call, for example rewards_claim({"reward_index":0}) MCP maps the typed tool call into the mod's REST action body.
9 MCP server STS2_MCP mod POST /api/v1/singleplayer Example body: {"action":"claim_reward","index":0}.
10 Harness JSONL log Trace record Logs state, model text, parsed JSON, tool result, and LLM usage. This is the best place to debug model behavior.

What The Model Sees

System Message

// Built once per model call from SYSTEM_PROMPT.
You are playing Slay the Spire 2 through MCP tools.

Game context:
- Slay the Spire is a roguelike deckbuilder...
- Each turn you draw a hand and spend energy...
- Deckbuilding goal: make a small, reliable deck...

Return exactly one JSON object and no prose:
{"action":"TOOL_NAME","args":{},"rationale":"short public reason"}

Rules:
- Choose exactly one action per response.
- Only play cards whose hand entry has can_play=true.
- On rewards, use rewards_claim first.
- Do not request get_game_state.

User Message

// Rebuilt every step.
Step: 12

Available actions for the current state:
- rewards_claim(reward_index:integer):
  Claim a reward...
  State hint: valid reward_index values: 0:card
- wait: Do nothing briefly...
- stop: Stop the harness cleanly.

Recent action history:
11. monster: combat_end_turn {} -> ok

Current game state JSON:
{"state_type":"rewards","rewards":{"items":[...]},...}

Pick the next single action now.

The model does not see hidden game state, the screen image, or the full prior transcript. It sees the current JSON state, a short history window, and the current-state tool menu that the harness built.

Expected Model Output

{
  "action": "combat_play_card",
  "args": {
    "card_index": 2,
    "target": "NIBBIT_0"
  },
  "rationale": "Use playable damage on the low HP attacker."
}
  • action must be a current-state MCP tool name, or local wait / stop.
  • All tool parameters must live under args.
  • Targets use exact enemy entity_id strings from game state.
  • The harness asks for repair if JSON is malformed or illegal.
  • Reasoning is off by default in llama-server, so hidden chain-of-thought is not shown.

Current-State Tool Mapping

State Type Tools Shown To Model Dynamic Hints
menu, game_over menu_select Visible option names such as singleplayer, confirm, main_menu.
monster, elite, boss combat_play_card, combat_end_turn Playable hand indices, card names, enemy target ids, can_play flags.
rewards rewards_claim, or proceed_to_map only after no items remain Reward indices and reward types.
card_reward rewards_pick_card, optional rewards_skip_card Offered card indices, names, and skip availability.
map map_choose_node Next node indices and node types.
event event_advance_dialogue during dialogue, otherwise event_choose_option Unlocked option indices, titles, and descriptions.
rest_site, shop, treasure Screen action plus proceed_to_map when allowed Enabled rest options, affordable shop items, treasure relic indices.
card_select, bundle_select, relic_select Selection tools plus confirm/cancel/skip when state flags allow them Selectable indices and confirmation flags.
crystal_sphere crystal_sphere_set_tool, crystal_sphere_click_cell, crystal_sphere_proceed Available tool sizes, clickable cells, and proceed flag.

Where To Improve

State affordances Move the current-state action mapping into STS2_MCP itself as an allowed_actions field, so every client gets the same legal menu.
Decision quality Add a small policy layer that scores card rewards, map nodes, potion use, and lethal/block choices before asking the model.
Memory Persist run-level notes such as deck plan, boss, relic synergies, path risk, and failed actions instead of relying only on recent history.
Observability Build a log viewer over logs/*.jsonl so you can inspect state, prompt, raw model output, validation errors, and tool result step by step.
Recovery Track repeated invalid choices and temporarily mask actions or indices that just failed for the unchanged state.
Prompt size Summarize stable run data and show only screen-relevant slices of state to fit longer runs into the local context window.
